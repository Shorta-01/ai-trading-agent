## Purpose of this document

This is the top-level doctrine for the AI Trading Agent. Every functional, architectural, and code-health decision in the project must be consistent with this document. Where this document conflicts with any other doctrine, plan, or product note in the repo, this document wins until explicitly superseded by a later dated doctrine.

## 1. What this system is

The AI Trading Agent is a **full trading system**. It analyses markets, generates order suggestions, and submits approved orders to a broker (Interactive Brokers) for real execution.

It is not a research tool. It is not a paper-only sandbox. It is not a notification service. It is software that places real orders against real accounts when the user approves them.

## 2. The broker relationship: IBKR is the single source of truth

The system connects to one IBKR account at a time. That account is owned and configured by the user inside IBKR. The software does not maintain a parallel or shadow view of the user's positions, cash, or orders.

For everything related to the user's account — positions held, cash balances, open orders, parked orders, order fills, FX exposure, dividends, account events — IBKR's state is the truth. The system reads from IBKR and reflects what it sees. When IBKR's state changes, the system updates. When the system needs to express a user intent against the account (e.g. park an order), it does so by writing into IBKR via the IBKR API; the result is then visible by reading IBKR back.

Consequences:
- The system never invents portfolio data.
- The system never holds an authoritative version of an order that contradicts IBKR.
- The system's audit trail records what it asked IBKR to do and what IBKR reported back, but the *current state* is always *whatever IBKR currently says*.

## 3. Account modes: paper and real are first-class

The IBKR account the system is connected to can be a paper account or a real-money account. The choice is made by the user inside IBKR, by connecting the appropriate IBKR session. The software treats both modes identically:

- Same code paths
- Same calculations
- Same suggestion logic
- Same order flow
- Same audit trail

There are no "paper-mode-only" features. There are no "real-money-only" features. The only difference between the two modes is which IBKR account the system is talking to, and consequently whether orders execute against simulated or real markets.

The purpose of paper mode in this system is solely to verify that the software behaves correctly before pointing it at a real-money account. Paper mode is not an onboarding flow, not a permanent parallel environment, and not a long-term home for any user.

### 3.1 The mode badge

The dashboard must always show a visually unmistakable badge indicating the current mode:
- PAPER mode: clearly marked, distinctive colour (yellow or equivalent)
- REAL MONEY mode: clearly marked, distinctive colour (red or equivalent)

The badge must never be ambiguous, hidden, or styled subtly. The user must never have to think about which mode is active — a glance must answer it.

## 4. The order lifecycle: two grids, two approvals

The system uses a two-step order approval model. Every order passes through two distinct grids on the dashboard, each with its own approval action.

### 4.1 The suggested orders grid

This grid contains order proposals generated by the system. Each row is a fully-formed IBKR-shaped order ticket, pre-filled with the system's best values for every field. The user can:
- Approve the order as-is
- Edit one or more fields, then approve
- Reject the order, with an optional one-word reason (cash / risk / personal / other), logged silently for later pattern analysis

Approving a suggestion does **not** submit the order to the market. It creates the order in IBKR as a parked order, where it becomes visible in the IBKR todo grid (below).

Rejected suggestions never reach IBKR. They are recorded in the audit trail but do not affect the account.

### 4.2 The IBKR todo grid

This grid is a live view of IBKR's current order book. It shows all parked orders in IBKR, regardless of origin (system-originated or manually placed by the user inside IBKR). Each row reflects the current state in IBKR. The user can:
- Approve the order for submission to the market
- Edit fields, then approve (edits are pushed to IBKR via the modify-order API and re-validated)
- Cancel the order (deletes it from IBKR)

Approving an order from this grid **is** the submission. The order goes live to the market.

### 4.3 The two approvals are different actions

The first approval (suggested grid → IBKR todo grid) crosses the boundary *system → IBKR*. The second approval (IBKR todo grid → market) crosses the boundary *IBKR parked → IBKR live*. Each approval is logged independently in the audit trail with timestamp, user action, order state, and resulting IBKR response.

## 5. Order content: the system pre-fills, the user approves

When the system generates a suggestion, it does not say "buy ASML" in prose. It builds a complete IBKR-shaped order ticket with every field populated.

### 5.1 Quantity

The proposed quantity is computed by combining three layers in this order:

1. **Base size:** fractional Kelly sizing using the existing `kelly_sizing.py` logic, defaulting to 0.25 Kelly (not full Kelly). Derived from per-trade risk budget and a system-derived stop-loss level.
2. **Conviction scaling:** the base size is multiplied by the forecast's confidence score. High-confidence signals get full base; lower-confidence signals get proportionally less.
3. **Hard caps:** maximum percentage per single position, maximum percentage per sector, maximum percentage per asset class. Caps are absolute — they override layers 1 and 2 if breached.

The order ticket shows the resulting share count. The explanation icon must indicate which of the three layers was the binding constraint for this specific order.

### 5.2 Order type

The system picks the best order type per situation (market, limit, stop, bracket). The choice is determined by the suggestion's logic, not by a global default.

### 5.3 Limit price (for limit orders)

The system pre-fills limit prices using patient-liquidity logic:
- Buys: current mid-price minus a configurable discount (default 0.2%)
- Sells: current mid-price plus a configurable premium (default 0.2%)
- The discount/premium is per-asset configurable, higher for high-volatility assets, lower for low-volatility ETFs
- Forecast override: when the system's forecast-implied fair value disagrees materially with the mid-price (threshold to be defined), the forecast value overrides the patient-liquidity default

The explanation icon must show which logic was used (patient liquidity, forecast override) and why.

### 5.4 Time-in-force

The system pre-fills time-in-force as DAY for every order. Unfilled orders auto-cancel at market close; the next morning's evaluation re-proposes them if still warranted. GTC and GTD are available only via the "show advanced" toggle and are never the system's default.

### 5.5 Validation (hybrid)

Order field validation is hybrid:
- **Client-side, instant:** cheap rules (positive numbers, valid symbol, lot-size multiples, valid TIF values, etc.) checked in the browser as the user edits
- **IBKR round-trip, on Approve:** the full "would IBKR accept this" check runs once when the user presses Approve, via IBKR's `whatIfOrder` (or equivalent). Returns margin impact, commission estimate, and any rejection reason
- If IBKR rejects, submission is blocked and the rejection reason is shown to the user

This validation applies in both the suggested grid (first approve) and the IBKR todo grid (second approve, against the parked order).

## 6. Synchronisation: hybrid push/pull, IBKR-driven

The system stays in sync with IBKR using a hybrid model:

- **Primary path:** event-driven subscription to IBKR's order-update and account-update streams. Any change in IBKR — new order, modification, fill, cancellation, position change, cash change — pushes immediately to the system and on to the dashboard
- **Fallback path:** periodic polling whenever the event stream is disconnected, throttled, or otherwise unhealthy. Polling interval to be defined per stream type

The dashboard's system-health line must reflect which mode is active so the user can see at a glance whether updates are real-time or polled.

This sync model applies to all IBKR-sourced state: orders, positions, cash, fills, account events.

## 7. The morning evaluation: every day from scratch

Every day at 07:00 the system performs a full evaluation: refresh prices, refresh FX, refresh IBKR positions, recompute valuations, generate forecasts, build decision packages, populate the suggested orders grid for the day.

Yesterday's unactioned suggestions do **not** carry over. The system re-evaluates from the current market state. If a previously-suggested action is still warranted today, it appears as a fresh suggestion. If conditions have changed, it does not.

This rule is in deliberate tension with persistent (GTC) orders, which is why the system's default TIF is DAY (§5.4). The morning evaluation also reconciles against IBKR's actual order book: any orders still alive from previous sessions (rare, e.g. manually-placed GTC orders) appear in the IBKR todo grid as the live source of truth.

## 8. Throughout the day: silent, hourly, additive

Between morning evaluations, the system runs lighter evaluations every hour. New suggestions append silently to the suggested orders grid as the system identifies them. There are no notifications, no interruptions, no badges that demand attention. The user checks the dashboard when they choose to.

The IBKR todo grid updates in real time via the sync model (§6), not on an hourly cadence.

## 9. Suggestion recurrence: counter, no snooze

When the system proposes the same suggestion (same asset, same direction, similar size) on consecutive days, the suggestion shows a recurrence counter inline ("3rd time this week"). This is informational. There is no snooze, no "reject for a week" option. Every day the system evaluates fresh; if the suggestion is still warranted, it appears again.

Rejection is per-day and includes an optional one-word reason (cash / risk / personal / other), captured silently for later pattern analysis.

## 10. The dashboard: three areas, no clutter

The dashboard contains three areas and only three areas:

1. **Portfolio:** current positions sourced from IBKR, with position-level current values. No daily P&L on the dashboard, no daily change colouring, no totals delta. Performance review lives on a separate screen (§11).
2. **Watchlist:** assets being tracked but not held. Each item shows an inline "proximity to action" signal indicating how close it is to triggering a buy suggestion.
3. **Actions:** the suggested orders grid (§4.1) and the IBKR todo grid (§4.2). Each order shows a trust signal (forecast confidence + data freshness) and an explanation icon (clear, plain-language rationale for why the system proposes this order).

A single system-health line at the top of the dashboard reflects sync status, data freshness, and IBKR connectivity. Green when healthy (and visually quiet). Red when degraded (and visually loud).

When the actions area is empty, a one-line explanation states why ("no signals triggered today, watchlist stable, portfolio within target ranges"). Empty without explanation is forbidden.

No charts, no news feed, no sentiment ticker, no market overview, no notifications panel, no past-actions history on the dashboard. Those belong on dedicated screens.

## 11. Performance review: separate screen, professional metrics

Performance review lives on a separate screen, not on the morning dashboard. The user visits it when they choose to, not as part of the morning routine.

The performance screen shows:
- Time-weighted return vs benchmark
- Drawdown from peak
- Volatility / risk-budget usage
- Exposure breakdown (asset class, sector, currency)
- Portfolio chart
- Weekly and monthly views

Daily P&L is deliberately not surfaced as a headline metric, here or on the dashboard. The intent is to support deliberate weekly/monthly review, not to encourage daily emotional reactions.

## 12. Audit trail: complete, immutable, separate

Every state-changing event is recorded in the audit trail:
- Suggestion generated
- Suggestion approved / edited / rejected (with reason)
- Order parked in IBKR (system-originated or manual)
- Parked order edited
- Parked order approved for submission / cancelled
- Order filled (partial or full)
- Sync events
- Mode changes (paper ↔ real connection changes)

The audit trail is append-only and lives on a separate screen, not on the dashboard. Audit entries reference the IBKR-side identifiers wherever applicable, so any event can be traced back to its IBKR record.

The pre-existing rule "no advice without audit trail" (`AGENTS.md`) remains in force.

## 13. AI: explanation only, never execution

AI (LLM) usage is bounded by the rule **Python calculates, AI explains** (from the original `AGENTS.md`, retained).

The AI is used to:
- Explain to the user, in plain language, why the system proposed a given order (the explanation icon, §10)
- Summarise research evidence and forecast reasoning
- Translate technical analysis into the user's language (Dutch, Flemish register where appropriate)

The AI is not used to:
- Decide quantities, prices, order types, or any other order field
- Override risk rules or investment policy caps
- Approve or reject orders
- Modify IBKR state directly

Every AI output is schema-validated. AI may not override deterministic risk or sizing logic. The AI explanation budget cap (the existing `claude_ai_budget` infrastructure) remains in force.
